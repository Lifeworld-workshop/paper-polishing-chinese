# 中文学术论文语言润色

对中文学术论文进行语言层面的系统润色，目标是提升可读性、逻辑性与学术表达质量，并保持作者原意与论证结构。

## 常驻主干（默认加载）

1. **准确第一**：不改变原意，不篡改观点。
2. **规范为本**：遵循 GB/T 15834-2011（标点）与 GB/T 15835-2011（数字）。
3. **叙事优先**：输出可直接入文的连贯学术叙述，避免流程图口吻。
4. **问而不猜**：术语冲突、事实口径不清时先确认再改。
5. **硬规则先行**：先执行硬规则，再做风格润色。
6. **懒加载优先**：主干保持短小，细则按任务加载对应 references。

## 硬规则（必须执行）

1. **术语区分**：`assumption` 用“假设”，`hypothesis` 用“研究假说”，不得混用。
2. **词义区分**：
   - “阐释”用于内涵/机制/精神层面的解释。
   - “阐述”用于观点/过程/事实的系统展开。
3. **区域协同研究审慎表述**：
   - 表述“地方政府对市场有话语权”，不写“控制力”。
   - 产能配置可协同。
   - 交通与能源可讨论一体化（通常作为协同前提）。
   - 教育不宜轻言一体化，需说明现实约束。
4. **分歧连接词**：文献观点不一致时优先使用“然而、相反地、从另一方面来说、虽然如此”（不限于这四个）。
5. **比较研究量化**：A 与 B 对比不得只写“更高/更低”，必须给出可审计量化表达（如“高出 12%”“为其 2.3 倍”）。
6. **禁忌用语分级执行**：按 `references/prohibited-wording.md` 的 A/B 级规则强制处理。
7. **删除程序性免责句**：定稿中不得保留"若后续官方发布更新数据……不影响本文论证"及一切类似的程序性、免责性、自我辩护性句子。此类句子在润色阶段必须直接删除，不做保留或改写。
8. **政策脚注来源规范**：政策类脚注的 URL 与出处必须指向正式权威来源——国务院（www.gov.cn）、中央人民政府门户网、全国人大（www.npc.gov.cn）、新华社（www.xinhuanet.com）等。严禁使用驻外使馆镜像链接或其他非权威转载站点。发现此类链接时，必须替换为对应政策文件的官方原始出处；若无法确认官方 URL，应标注提示让作者核实。
9. **"创新点"→"边际贡献"**：将"创新点/创新之处"统一替换为"边际贡献"或"相对于现有研究的边际贡献"，条目标题中的"创新"同步替换。
10. **本文/本章严格区分**：章节内部描述本章操作、样本、结论时用"本章"；"本文"仅用于全文层面（引言、摘要、贡献说明）。

## 主干高频词例（保留 15 条）

| 词/短语 | 推荐用途 | 约束 |
|---|---|---|
| 矛盾激化 | 描述冲突升级 | 需有时间趋势或证据支撑 |
| 随着……改革推进 | 引入阶段变化 | 后句需接机制或结果 |
| 丧失优势 | 描述相对地位变化 | 建议补充比较对象 |
| 发展陷入困境 | 描述发展受阻 | 建议说明约束来源 |
| ……成为……的关键因素 | 强调核心机制 | 避免泛化空喊 |
| 打破……壁垒 | 描述制度/市场摩擦下降 | 说明“壁垒”类型 |
| 为……奠定基础 | 描述前置条件 | 后文需承接具体结果 |
| 弊端日益凸显 | 描述问题显性化 | 需给事实或案例 |
| 生产效益低下 | 描述效率不足 | 建议附指标口径 |
| 资本转化为新的流通形式 | 描述资本形态演化 | 注意与制度背景匹配 |
| 阻碍 / 抑制 / 制约 | 描述负向作用 | 区分强度与作用路径 |
| 促进 / 改善 / 助推 / 推进 | 描述正向作用 | 优先与可观察结果对应 |
| 优化 / 健全 / 完善 / 提升 | 描述政策与治理改进 | “建立”仅用于从无到有场景 |
| 研究……与……的关系 | 学术问题陈述 | 避免“对……进行阐述”套话 |
| 探索……的路径 | 对策与机制分析 | 避免路径泛化 |

## 工作流程

### 步骤 1：识别任务类型

- 全文润色
- 局部润色
- 英译中母语化润色
- 格式规范化（标点、数字、引用）

### 步骤 2：最小化加载 references

按任务仅加载必要文件，不全量读取。

### 步骤 3：执行润色

1. 先跑硬规则核查（术语、比较量化、分歧连接词、禁忌用语、程序性免责句删除、政策脚注来源规范、创新点→边际贡献、本文/本章区分）。
2. 再进行语言层润色（去口语化、句式、衔接、冗余削减）。
3. 默认只输出可直接入文版本；仅在用户要求时输出逐条台账。

### 步骤 4：生成修订稿（如需 .docx）

.docx 修订稿必须走 `docx` 技能原生流程（禁止视觉模拟修订脚本）。

```bash
# 1) 解包原始文档
python scripts/office/unpack.py /path/to/input.docx unpacked/

# 2) 在 unpacked/word/document.xml 中写入原生修订标签
#    使用 <w:del>/<w:ins>；仅标注最小必要改动

# 3) 回包生成修订稿
python scripts/office/pack.py unpacked/ /mnt/user-data/outputs/润色修订稿.docx --original /path/to/input.docx
```

## 关键提醒

- 润色只改语言，不改学术立场与论证结论。
- 任何“硬规则冲突”优先提示并修正。
- 标题相关模板不作为本技能规则内容。
